# Phase 4.5 — Consistency Gate

**Goal:** Verify structurally that the wave-plan covers the spec before Phase 5
(Implement) starts. This phase is a hard gate: on FAIL, do not advance to
Phase 5.

**Sub-agent:** none. The gate is a deterministic CLI call.

**Status transition:**
- On PASS: `planned` (unchanged) — proceed to Phase 5.
- On FAIL: `planned` → `awaiting-consistency-fix`. Phase 5 does **not** start.
- On ERROR: `planned` (unchanged). Phase 5 does **not** start. User must
  repair artifacts.

## Precondition

- Spec status is `planned`.
- Wave-plan file exists at `projects/<slug>/plans/<###>-<feature-slug>-wave-plan.md`.
- Spec file exists at `projects/<slug>/spec/<###>-<feature-slug>.md`.

If either file is missing, fall back to Phase 4 (Plan) before reaching this gate.

## Step 1 — Run the gate (JSON mode)

Use the **Bash** tool:

```bash
node ~/.claude/skills/_shared/a1-tools.cjs check <project-slug> \
  --feature <###>-<feature-slug> --format json
```

Capture stdout and exit code. Do **not** parse stderr for control flow —
non-zero exit is the signal.

## Step 2 — Branch on exit code

### Exit 0 — PASS

Tell the user (German):

> "Konsistenz-Gate bestanden: Spec und Wave-Plan sind synchron. Phase 5
> (Implement) kann starten. Soll ich Wave 1 dispatchen?"

Then load `workflows/05-implement.md` on confirmation.

### Exit 1 — FAIL (content inconsistency)

1. Re-run the CLI with `--format human` and show the German report verbatim
   to the user.
2. Update the spec status to `awaiting-consistency-fix`:

   ```bash
   node ~/.claude/skills/_shared/a1-tools.cjs spec update-status \
     "projects/<slug>/spec/<###>-<feature-slug>.md" awaiting-consistency-fix
   ```

3. Parse the JSON output to derive the fix-path suggestion (same table as in
   `a1-check/workflows/01-run-check.md` Step 3):

   | Diff field non-empty | Suggested next phase |
   |---|---|
   | `missing_in_plan` or `duplicated_in_plan` | Phase 4 (Plan) — Vincente reworks the wave assignments |
   | `phantom_in_plan` | Phase 3 (Clarify) — Rene adds the missing FRs, or user edits the plan |
   | `checks.frontmatter_link == "FAIL"` | Targeted edit of plan frontmatter `spec_path` (no full phase re-run) |

4. Ask the user (German) **one** question with the matching suggestion. Do
   **not** auto-trigger any phase. Wait for confirmation.

5. On confirmation: route to the named phase (or perform the targeted edit).
   After the fix, Phase 4.5 re-runs automatically because the spec status is
   `awaiting-consistency-fix` and the skill routing brings the user back here.

### Exit 2 — ERROR (setup)

Show the human report verbatim (stderr from the CLI explains which file is
missing). Do **not** change the spec status — the spec stays `planned` so
that on re-entry the routing leads naturally back to Phase 4.5 once the
missing artifact is in place.

Tell the user (German):

> "Setup-Problem: Eine der nötigen Dateien fehlt oder ist defekt (Details
> oben). Bitte beheben — danach erneut `a1-new-feature` aufrufen, das Gate
> läuft automatisch wieder."

Stop.

## Step 3 — Re-entry routing (when status is `awaiting-consistency-fix`)

When the user later re-invokes `a1-new-feature` and the spec status is
`awaiting-consistency-fix`, the skill jumps directly to Phase 4.5 (this
workflow). Do **not** restart Phase 1–4.

The routing logic (in `SKILL.md` "Routing — pick the right phase") handles
this status: `awaiting-consistency-fix` → Phase 4.5.

## Hard rules

- Never bypass this gate. Phase 5 must not start on FAIL or ERROR.
- Never edit the wave-plan or spec from this workflow. The fix must come
  from a re-entered phase (Plan or Clarify) or a deliberate user edit.
- The gate is read-only and deterministic — do not spawn sub-agents to
  "explain" the failure. The CLI's `--format human` output is the
  explanation.
- Spec status updates here use the helper (`a1-tools spec update-status`),
  never raw frontmatter edits.

## Hand-off

- PASS → `workflows/05-implement.md`
- FAIL (after user confirms re-entry) → relevant earlier phase workflow
- FAIL (user declines re-entry) → stop, spec remains in `awaiting-consistency-fix`
- ERROR → stop, spec remains in `planned`
